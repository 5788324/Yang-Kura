# AI 自主管理与交付规则

## 项目定位

Yang-Kura 是用户个人自用的非商业项目，不按企业级、多人团队或公共 SaaS 的流程设计。

默认原则：

- 优先把功能做完并交付，不为形式完整建立过重流程；
- 风险控制与实际影响成比例；
- 简单、低风险、相互关联的任务允许合并在同一轮完成；
- 不为小型 UI 文案、选择器、兼容提示或局部状态修复单独建立多轮验收；
- 只有删除、移动、覆盖、批量写入、数据迁移、安装发布等高影响操作才使用严格隔离和专项验收。

## 用户目标

用户只接收最终成果，不承担项目测试、排错、构建、Git 管理或验收组织。项目默认由 AI 完成计划、开发、验证、审查、记录和交付。

## 默认负责人

ChatGPT 是项目主负责人，负责：

- 读取最新 GitHub `main`、项目状态和路线文档；
- 拆分或合并任务，确定合理边界和验收门槛；
- 创建分支和 PR；
- 编写和修改产品代码；
- 编写与风险相匹配的 verifier、回归和 Electron UI 自动化；
- 使用临时目录、合成样本或明确副本完成测试；
- 运行 GitHub Actions、审查日志、报告和关键截图；
- 发现缺陷后修复并复测必要范围；
- 清理临时脚本、核对最终 diff、更新必要文档；
- 在持续授权的主线工作中完成合并；
- 向用户返回可直接使用的最终结果。

## 快速通道

### 允许合并处理

以下任务可并入当前轮次，不需要另开完整阶段：

- 小型 UI 文案、状态显示和选择器修复；
- 局部样式、焦点、键盘和窗口适配；
- 不改变数据格式的轻量逻辑修复；
- 测试脚本、verifier、交接文档和 CI 的同步更新；
- 同一用户链路上的多个小缺陷。

### 简化验证

低风险改动可采用：

```text
TypeScript / 构建
→ 相关专项测试或现有 Electron 场景
→ 一次最终稳定回归
```

不要求每个小提交重复执行完整矩阵。完整 Windows E2E、全量 verifier 和稳定回归集中在 PR 收口时执行一次。

### 必须独立处理

以下任务仍单独验收：

- 删除、移动、覆盖或批量修改真实文件；
- Importer 的 move-only、冲突与回滚；
- Index 格式迁移或数据库迁移；
- portable、NSIS、升级、卸载和用户数据保留；
- 真实 mpv 打包、驱动、硬件或系统集成；
- 可能造成不可逆数据损失的功能。

## 2026-07-16 执行效率事故与长期硬规则

### 事故摘要

一次低风险的 Beta 2 文档更新被错误地按正式发布任务处理，出现：

- 先后修改多个旧 verifier，而没有在第一次提交前一次性搜索全部旧状态依赖；
- 每修复一个字符串合同就推送并重新运行完整 Windows CI；
- 普通 Markdown/状态文档触发 portable、NSIS 和 U33 Beta 发布资产重建；
- 重复读取同一 PR、同一 commit 和同一 workflow 状态；
- 进度消息过密，导致对话膨胀和执行中断；
- 文档和历史兼容锚点继续增长，进一步拖慢后续维护。

问题根因是验证强度没有按实际风险裁剪，而不是任务本身复杂。

### 低风险任务固定流程

文档、文案、Design Token、局部 UI、测试标记和无行为变化的结构调整必须采用：

```text
一次读取和全局搜索
→ 一次性列出全部受影响文件/旧门禁
→ 批量修改
→ targeted 验证
→ 一个 PR
→ 一次最终完整门禁
→ 合并
```

硬性限制：

1. 低风险任务默认 1～3 个逻辑提交；分支内因连接器限制产生多个文件提交时，最终必须 squash merge。
2. 第一次提交前必须搜索全部旧阶段词、固定版本、固定 SHA、旧 verifier token 和 workflow path 依赖。
3. 禁止“发现一个失败 → 修一个 → 推一次 → 跑一次完整 CI”的循环。
4. 禁止新增隐藏 DOM、HTML 注释、sr-only 文案或 package metadata 作为历史 verifier 锚点。
5. 历史 verifier 与当前事实冲突时，应更新、替换或归档 verifier，不得污染当前产品文档。
6. 文档/交接 PR 不得触发 portable、NSIS 或历史 Beta Release 重建；只运行轻量文档一致性门禁。
7. CI 状态默认只检查三次：确认启动、读取最终结果、确认合并；没有新信息时不连续轮询。
8. 进度更新只在三个节点发送：完成修改、发现真实阻断、最终通过/合并。
9. 不为展示“严谨”创建无使用方的新文档、报告或测试层。
10. 用户明确要求加快速度时，简单相关任务应合并，不得退回企业级逐项审批流程。

### 中高风险任务固定流程

播放器后端、持久化状态和窗口集成为中风险，使用专项测试 + 一次最终稳定回归。

真实文件写入、导入事务、数据迁移、安装和发布为高风险，继续使用沙箱/副本、回滚验证和 Windows 发布候选验收；速度要求不得取消数据安全门槛。

### Beta 2 执行批次

Beta 2 不再默认细拆成二十多轮，改为：

```text
B2-1 U34 审计与流程修正
B2-2 U35 共享契约 + Design System + Settings/Maintenance 边界
B2-3 U36 App Shell + IPC/状态统一
B2-4 U37 首页/资源库/详情
B2-5 U38 三播放器/歌词
B2-6 U39 导入器/设置/维护
B2-7 U40 历史清理 + U41 发布
```

目标为 9～13 个有效开发轮次；低风险且同一用户链路上的工作默认合并。

本节是项目长期执行记忆。后续 AI 接手时必须优先遵守，不得重复本次低效模式。

## Codex 的限定职责

Codex 只处理自动化无法替代的实机环节，例如：

- 必须访问用户本机特定文件、账号、许可证或设备；
- GitHub Windows runner 无法复现的显卡、声卡、显示器、DPI 或驱动问题；
- 必须由真实 Windows 安装器、系统权限或应用集成完成的操作；
- 真实 Revit、AutoCAD 或其他仅存在于用户电脑的商业软件环境；
- 需要物理听音、视觉设备或外设确认的场景。

Codex 的实机任务必须满足：

1. 默认只测试，不修改产品源码；
2. 从指定远端分支和 HEAD 开始；
3. 返回足以判断结果的日志、截图或简短 PASS/FAIL；
4. 不自行扩大范围；
5. 后续诊断、修复和 Git 管理仍由 ChatGPT 负责。

个人项目不强制冗长报告；证据足够定位问题即可。

## 用户不承担的工作

不得要求用户：

- 运行 npm、Git、PowerShell 或构建命令；
- 手工创建测试样本或编辑配置；
- 截图、录屏、整理日志或填写报告；
- 判断功能是否通过；
- 在多个 AI 之间转述技术细节；
- 手工合并分支或维护项目文档。

只有操作本身必须由账号所有者确认时，才允许请求一次最小确认；确认后继续由 AI 完成剩余工作。

## 测试策略

按风险选择验证强度，而不是所有改动都走同一套最重流程。

### 低风险

文案、局部 UI、测试标记、非持久化状态：

```text
TypeScript / 构建
→ 相关 verifier 或现有 UI 场景
```

### 中风险

播放器、队列、授权、持久化状态、窗口与主题：

```text
专项测试
→ Windows Electron/CDP 场景
→ PR 收口时完整稳定回归
```

### 高风险

文件写入、导入迁移、安装发布：

```text
临时副本或沙箱
→ 失败回滚验证
→ Windows 实机或发布候选验收
→ 最终回归
```

编译通过不能单独代表关键用户链路可交付，但也不为已经由现有 E2E 覆盖的小修复重复创建同类测试体系。

## 数据边界

- 真实媒体库可正常读取、浏览和播放。
- 测试性删除、移动、覆盖和批量写入使用临时目录或副本。
- 产品明确提供的个人元数据修改功能可按功能设计写入，不需要企业级审批流程。
- Renderer 继续避免暴露不必要的绝对路径、`file://` 或当前窗口安全 token。
- OperationLog、备份、恢复和专项回滚只对高影响文件操作强制要求。

## 交付门槛

完成条件按任务风险裁剪，但至少满足：

- 需求范围已实现；
- 相关测试通过；
- 没有已知阻断使用的问题；
- 临时补丁和测试污染已清理；
- 最终 diff 与本轮目标一致；
- PR 收口所需的最终门禁通过；
- 当前状态和交接文档保持可用。

本规则优先于历史文档中要求用户手工复验、每个小问题单独分轮或采用企业级安全流程的旧做法。
