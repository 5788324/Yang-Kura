# AI 自主管理与交付规则

## 用户目标

用户只接收最终成果，不承担项目测试、排错、构建、Git 管理或验收组织。项目默认由 AI 完成计划、开发、验证、审查、记录和交付。

## 默认负责人

ChatGPT 是项目主负责人，负责：

- 读取最新 GitHub `main`、项目状态和路线文档；
- 拆分任务、确定边界和验收门槛；
- 创建分支与 Draft PR；
- 编写和修改产品代码；
- 编写专项 verifier、稳定回归和 Electron UI 自动化；
- 使用临时目录和合成样本完成安全测试；
- 运行 GitHub Actions、审查日志、报告和截图；
- 发现缺陷后继续修复并从头复测；
- 清理临时脚本、核对最终 diff、更新文档；
- 标记 Ready、提出合并结论并在获得持续授权的主线工作中完成合并；
- 向用户返回可直接使用的最终结果。

## Codex 的限定职责

Codex 只处理自动化无法替代的实机环节，例如：

- 必须访问用户本机特定文件、账号、许可证或设备；
- GitHub Windows runner 无法复现的显卡、声卡、显示器、DPI 或驱动问题；
- 必须由真实 Windows 安装器、系统权限、原生文件对话框或应用集成完成的手工操作；
- 真实 Revit、AutoCAD 或其他仅存在于用户电脑的商业软件环境；
- 需要物理听音、视觉设备或外设确认的场景。

Codex 的实机任务必须满足：

1. 只测试，不修改产品源码；
2. 从指定远端分支和 HEAD 开始；
3. 使用明确的只读/可写边界；
4. 返回结构化证据、日志、截图和 PASS/FAIL；
5. 不自行合并、不扩大范围；
6. 后续诊断和修复仍由 ChatGPT 负责。

## 用户不承担的工作

不得要求用户：

- 运行 npm、Git、PowerShell 或构建命令；
- 手工创建测试样本或编辑配置；
- 截图、录屏、整理日志或填写报告；
- 判断功能是否通过；
- 在多个 AI 之间转述技术细节；
- 手工合并分支或维护项目文档。

只有操作本身必须由账号所有者确认时，才允许向用户请求一次最小确认；确认后继续由 AI 完成剩余工作。

## 测试优先级

验证顺序：

```text
纯领域逻辑测试
→ 专项 verifier
→ TypeScript / Electron 构建
→ 临时样本集成测试
→ Windows Electron/CDP 全链路 UI 测试
→ 截图视觉审查
→ 完整稳定回归
→ 二次生产构建
→ 最终 diff 与仓库卫生检查
```

编译通过、字符串 verifier 通过或状态对象正确，均不能单独代表可交付。关键用户链路必须验证最终 DOM、真实后端状态和页面截图。

## 数据安全

- 真实媒体库默认只读。
- 可写测试只使用临时目录、合成样本或明确副本。
- 不删除、移动、覆盖、重命名或批量修改用户真实文件。
- Renderer 不暴露绝对路径、`file://` 或当前窗口安全 token。
- 高风险操作必须有独立任务、OperationLog、备份/恢复和失败回滚。

## 交付门槛

只有同时满足以下条件，才能向用户报告完成：

- 需求范围已实现；
- 专项测试和完整回归通过；
- 关键 UI 截图已由 AI 审查；
- 无未解释 Blocker/Major；
- 临时补丁、诊断脚本和测试污染已清理；
- 最终 PR diff 与任务边界一致；
- 最终 HEAD 的永久门禁通过；
- 项目状态、路线和验收文档已更新。

本规则优先于历史文档中要求用户手工复验或转述测试结果的旧流程。
