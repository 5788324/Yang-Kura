# Yang-Kura Beta 2 总体计划

> 文档状态：执行基线 1.0  
> 当前稳定发布：`0.168.0-beta.1`  
> Beta 2 目标版本：`0.169.0-beta.2`  
> 适用范围：Beta 1 发布完成后到 Beta 2 发布完成前  
> 代码事实来源：GitHub `main`

## 1. 阶段结论

Yang-Kura 已完成 Beta 1 主线和 Windows 发布闭环。当前项目已经具备本地音声库、音乐库、播放、字幕、导入、元数据、资源库维护和 Windows 安装包等核心能力。

Beta 2 的目标不是增加新的大型业务功能，而是把现有软件从“功能完整的 Beta”提升为“结构可维护、视觉统一、适合长期日用和继续扩展的产品基线”。

Beta 2 统一处理两类欠账：

1. 代码结构、模块边界、类型、状态、IPC、测试和文档治理；
2. App Shell、页面布局、主题材质、动画系统、三种播放器和真实功能入口的 UI 重写。

这两类工作必须在同一计划中推进。禁止先做一套脱离真实业务的漂亮壳，也禁止先进行长期不可见的全项目重构。

## 2. Beta 2 范围

### 2.1 必须完成

- 建立代码库架构与质量基线；
- 明确 renderer、preload、Electron main、domain、application、infrastructure 的依赖方向；
- 建立统一 IPC、错误和 Result 契约；
- 建立正式设计系统和语义化主题 Token；
- 完成深色“暮夜琥珀”和浅色“雾光象牙”；
- 重写 App Shell、导航、顶栏、搜索和页面网格；
- 重写首页、音声库、作品详情、音乐库、歌单、导入器、设置和 AI 维护页面；
- 完成经典、黑胶、歌词三种播放器；
- 补齐现有真实功能对应的按钮、菜单、空状态、错误状态和恢复流程；
- 建立可阻止架构和视觉退化的 CI 门禁；
- 验证 Beta 1 到 Beta 2 的设置、Index、歌单、历史、元数据和用户数据兼容；
- 生成 portable、NSIS 和 SHA-256 发布资产。

### 2.2 明确后置

Beta 2 完成前禁止启动或合入：

- 正式下载器；
- SQLite 全面迁移；
- OpenList / WebDAV / 云同步；
- 新元数据 Provider 大扩展；
- Player Core v2 或更换播放器内核；
- 转录 Worker 正式集成；
- 插件市场或通用插件平台；
- 大型新业务页面；
- 全量文件格式或媒体服务器生态扩展。

允许修复与这些模块相关的 P0/P1 回归，但不允许借修复扩大范围。

## 3. 执行阶段

## U34：审计、指标和不可破坏行为

目标：先建立可信基线，不修改产品行为。

交付：

- `docs/ARCHITECTURE_AUDIT.md`；
- 模块与依赖图；
- 大文件、复杂函数、循环依赖、重复代码和弱类型清单；
- UI 功能入口矩阵；
- 构建、测试、包体、启动和大库性能基线；
- Beta 1 不可破坏行为清单；
- 分阶段重构 Backlog。

验收：

- 现有稳定回归全部通过；
- 不改变 UI 和真实数据行为；
- 每个待重构模块有风险、依赖、迁移和回退说明。

预计：1～2 轮。

## U35：设计系统与应用壳

目标：建立新 UI 的稳定地基。

交付：

- 语义化 Design Token；
- 深浅主题完整材质系统；
- 字体、间距、密度、圆角、边框、阴影和图标规则；
- Motion 动效规则和 reduced-motion；
- 基础组件：Button、Dialog、Drawer、Menu、Tabs、Toast、Empty、Skeleton；
- 新 App Shell、侧栏、顶栏、全局搜索和底部 PlayerBar 容器。

验收：

- 主题不是简单换色；
- App Shell 不承载业务逻辑；
- 键盘、焦点、Escape、DPI 和窗口矩阵通过；
- 旧页面可暂时在新 Shell 内运行，允许渐进迁移。

预计：2～3 轮。

## U36：契约、状态和模块边界

目标：让 UI 重写建立在可维护的业务接口上。

交付：

- `shared/contracts` 统一类型；
- IPC channel 集中注册和运行时校验；
- 统一 Result 与错误分类；
- 资源库、播放器、导入器、设置和 Provider 的 application service；
- 清理 UI 直接访问文件系统、裸 IPC、重复状态和跨模块内部状态读取；
- ESLint/import 边界规则。

验收：

- domain 不依赖 React、Electron 或 Node 文件系统；
- renderer 不出现裸 IPC 字符串；
- 新代码不得增加 `any`、无解释断言和浮动 Promise；
- 同一业务状态只有一个事实来源。

预计：2～3 轮。

## U37：首页、资源库和详情迁移

目标：迁移最高频的内容浏览流程。

范围：

- 首页；
- 音声库网格、列表、高级筛选和批量操作；
- RJ 作品详情、音轨、字幕、附属文件、元数据和 Provider 入口；
- 音乐库、专辑详情、艺术家详情和音轨表格；
- 首次启动、扫描、Index 读取、损坏恢复和空状态。

验收：

- 所有现有真实能力都有用户可理解的入口；
- 工程信息不进入日常层；
- 50,000 曲目基准不明显退化；
- 列表虚拟化、图片懒加载和筛选响应达到性能预算。

预计：3～4 轮。

## U38：三种播放器与播放工作流

目标：重做播放器视觉和交互，不更换现有 mpv + HTMLAudio 内核。

范围：

- 常驻 PlayerBar；
- 经典播放器；
- 黑胶播放器；
- 歌词播放器；
- 队列抽屉；
- 睡眠定时；
- 播放完成策略；
- 字幕语言、时间偏移、来源和重新关联；
- mpv fallback 的非阻断反馈；
- ASMR 长音频和作品结构入口。

验收：

- 三种模式共享同一播放状态，不产生双重状态源；
- 全屏展开、关闭、焦点返回和 Escape 正常；
- reduced-motion 下关闭黑胶持续旋转和非必要转场；
- 长音频 Seek、恢复、退出回收和 fallback 回归全部通过。

预计：2～3 轮。

## U39：导入、设置、AI 维护与二级流程

目标：迁移复杂但低频的真实工作流。

范围：

- 导入来源、识别预览、冲突处理、确认、执行、失败和回滚；
- copy-only / move-only 的安全提示；
- 元数据编辑、字段对比、选择性应用和历史恢复；
- 资源库目录、播放、字幕、外观和导入设置；
- 缺失文件、备份恢复、Index 清理、Provider、mpv 和导入日志的 AI 维护页面。

验收：

- 不自动覆盖、删除或清理真实媒体文件；
- 操作前有预览，操作后有摘要，失败时有可理解的恢复入口；
- absolutePath、file://、原始 IPC 和内部错误只存在于受控维护层；
- 导入事务和回滚矩阵全部通过。

预计：2～3 轮。

## U40：质量收口与 Beta 2 发布

目标：删除旧 UI，完成长期治理和正式发布。

交付：

- 删除已迁移页面的旧实现和无用 CSS；
- 归档一次性 U33 发布工作流和历史活动文档；
- 把历史 MVP verifier 与日常稳定门禁分层；
- 完成视觉回归、无障碍、性能、安装升级卸载和数据兼容测试；
- 更新 README、状态、路线图、交接、Release Notes；
- 发布 `v0.169.0-beta.2`。

验收：

- Branch Validation 全绿；
- portable 和 NSIS 安装链全绿；
- Beta 1 用户数据升级到 Beta 2 后可读取；
- 主要页面深浅主题、三档窗口和 DPI 截图通过；
- 无 P0/P1 已知问题；
- Release tag、资产、大小和 SHA-256 回读一致。

预计：2～3 轮。

## 4. 总工作量

预计 12～18 个开发轮次。

轮次不是固定日期。每轮必须满足：

1. 范围单一且可回退；
2. 自动测试通过；
3. 真实功能不回归；
4. 文档与代码同步；
5. PR 审查后再合入；
6. 不把未迁移旧代码和新代码长期并行保留。

## 5. Beta 2 发布定义

Beta 2 完成不等于所有长期功能完成。它表示：

- 现有本地媒体库功能足以长期日用；
- UI 已完成系统性重写，不再是换色主题和零散动画；
- 代码边界和测试允许继续安全扩展；
- Windows 安装、升级、卸载和用户数据兼容可靠；
- 后续大功能可以通过明确扩展点接入。

Beta 2 可以作为第一版正式面向日常使用的公开 Beta。建议发布后进行 2～4 周真实使用观察；无 P0/P1 问题后，再决定进入 RC / 1.0，或启动下载器、SQLite、转录等下一阶段。

## 6. 决策原则

- 不推倒重写；
- 不同时维护两套完整 UI；
- 不为了抽象而抽象；
- 不引入微服务或复杂依赖注入框架；
- 不追求无意义的 100% 覆盖率；
- 不以历史 MVP 字符串校验阻碍当前产品演进；
- UI 设计以 `docs/DESIGN.md` 为最高视觉基线；
- 架构工作以 `docs/ARCHITECTURE_QUALITY_PLAN.md` 为执行基线；
- 当前进度以 `docs/PROJECT_PROGRESS.md` 和根目录 `PROJECT_STATE.md` 为准。
