## 3. 当前主要欠账

### 代码结构

- 需要真实审计超大组件、Hook、Electron main 和 service；
- 模块依赖方向尚未形成自动门禁；
- IPC、错误、Result 和数据类型需要统一；
- 部分状态与持久化可能仍分散；
- 历史 MVP/U verifier 与当前稳定门禁需要分层；
- 活跃源码中的 legacy 文件需要分类和逐步清理。

### UI 和设计

- 现有界面层级仍带工程面板感；
- 主题主要表现为颜色变化，缺少材质、排版和组件差异；
- 动画规则不统一；
- 页面之间视觉语言不够一致；
- 三种播放器仍需要完整设计和真实状态整合；
- 部分真实功能入口藏得过深或缺少二级流程；
- 首次启动、扫描、错误恢复、批量操作、元数据字段对比、导入失败和回滚详情需要重做。

### 发布和维护

- 顶层状态文档仍保留 U33 进行中文案；
- 一次性 U33 发布工作流需要在 Beta 2 收口时归档或改为按需运行；
- Beta 1 到 Beta 2 的用户数据升级矩阵尚未建立；
- 视觉回归和截图基线尚未成为稳定门禁。

## 4. 当前优先级决策

### 不是代码或 UI 二选一

正确顺序：

```text
架构和质量审计
→ 设计系统与新 App Shell
→ 契约和状态边界
→ 分页面迁移真实功能
→ 三种播放器
→ 质量、性能、兼容和发布收口
```

原因：

- 只先重构代码，用户长期看不到收益，且可能为不存在的问题设计抽象；
- 只先重写 UI，容易再次把真实业务逻辑堆进新页面，形成第二套技术债；
- 设计系统和模块边界同时建立，可以让每迁移一个页面就顺手完成相关结构整理，并立即删除旧路径。

### 近期执行顺序

1. U34 代码库和 UI 联合审计；
2. U35 Design Token、基础组件和 App Shell；
3. U36 契约、状态和 IPC 边界；
4. U37 首页、音声库、详情、音乐库；
5. U38 PlayerBar、经典、黑胶、歌词、队列；
6. U39 导入器、设置、AI 维护和复杂二级流程；
7. U40 清理、视觉回归、性能、升级兼容和 Beta 2 发布。

## 7. Beta 2 后的安排

Beta 2 可以作为第一版真正面向长期日用的公开 Beta。建议：

1. 发布后保持 2～4 周观察期；
2. 只修复真实使用发现的 P0/P1/P2 问题；
3. 检查大库、长音频、字幕、导入、安装升级和异常恢复；
4. 观察期结束后召开下一阶段规划；
5. 再决定优先启动下载器、SQLite、转录、元数据中心或其他功能；
6. 无重大问题时进入 RC 或 1.0 规划。
