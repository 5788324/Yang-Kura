# Yang-Kura 日常界面硬规则

## 长期原则

> 日常层只展示用户实际会使用的功能；诊断、回归、工程状态、测试入口、命令行说明、MVP/版本收口信息和检修工具统一进入 AI 维护或隐藏兼容层，不得长期污染主界面。

该规则是项目级长期硬规则，不随单个 U 轮次结束而失效。

## 默认可见层允许内容

日常界面只允许直接呈现：

- 媒体浏览、搜索、播放、字幕、队列、收藏和歌单；
- 资源库选择、读取、扫描和导入等用户动作；
- 主题、播放偏好、目录、隐私和其他真实设置；
- 与当前操作直接相关的错误、进度、结果和恢复提示。

## 默认不可见内容

以下内容不得作为常驻卡片、窗口或一级导航展示：

- Diagnostics、Runtime Probe、组件健康、回归状态和验收清单；
- MVP 编号、Beta/RC 收口、GitHub 基线、交接状态和发布流水线；
- npm 命令、环境变量、测试音频要求和开发者操作步骤；
- scanner contract、Allowed/Blocked、dry-run 协议解释等工程合同；
- 仅用于 verifier 的历史说明卡、芯片和兼容标记。

## 资源库设置分层

资源库页面必须明确区分日常操作与检修操作：

- 默认可见：选择目录、查看上次资源库、读取已有记录、一键扫描并应用、管理音声库和音乐库路径；
- 进入 `AI 维护`：缺失文件检查、失效索引清理、索引备份恢复、维护历史、扫描预览、写入预览和其他高级工具；
- 检修功能可以保留完整能力，但必须默认折叠，不得与快速导入并列抢占日常视觉层级；
- 历史回归卡、开发合同和版本记录即使位于维护区域，也应优先使用隐藏兼容层，而不是继续显示为操作面板。

## AI 维护边界

- 有实际排障价值的功能应保留，但默认折叠或集中到 `AI 维护`。
- 日常页面可以提供简短、用户可理解的状态；详细诊断数据应由 AI 维护页读取。
- 不得为了界面干净直接破坏诊断路由、测试合同、日志或底层修复能力。
- 历史 verifier 必须保留时，使用 `hidden` 与 `aria-hidden="true"`，不得重新变成可见产品 UI。

## 新增 UI 审查问题

任何新窗口、卡片或入口合入前必须回答：

1. 用户日常会直接操作它吗？
2. 它是否帮助完成当前用户任务，而不是解释项目如何开发？
3. 若只用于排障，为什么不能进入 AI 维护？
4. 是否把内部术语、版本号或测试步骤暴露给普通用户？

只要第 1～2 项不能明确成立，该内容默认不得进入日常层。
